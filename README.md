# Feedbook Backend

REST API for the feedbook app. NestJS + TypeScript, PostgreSQL, Drizzle ORM, Redis, Cloudflare R2.

**Live:** [feedbook.up.railway.app](https://feedbook.up.railway.app)  
**API Docs:** [feedbook.up.railway.app/docs](https://feedbook.up.railway.app/docs)

---

## Stack

- **NestJS 11** — modular framework with DI
- **Drizzle ORM** — SQL-like query builder, no generated client
- **PostgreSQL 17** — UUIDv7 PKs, composite indexes for cursor pagination
- **Redis 7** — session cache and rate limiting (falls back to in-memory map when Redis is down)
- **Cloudflare R2** — S3-compatible object storage for images
- **Argon2id** — password hashing at timeCost=3, mem=65536, parallelism=4
- **class-validator** — DTO validation everywhere

```
    ├── likes/               # toggle + get likers for posts/comments/replies

```

## Auth

Session-based. No JWTs.

On register or login we generate a random 64-character hex string, SHA-256 hash it, store the hash in `sessions.token_hash`, and hand the raw token back to the client. The client sticks it in an httpOnly cookie plus sends it as `Authorization: Bearer <token>` on every request.

AuthGuard hashes the incoming token, checks Redis (300s TTL), falls back to the DB on a cache miss. Logout deletes the row from `sessions` and clears the Redis entry.

Sessions mean you can revoke a token instantly without any blacklist handling.

## Database

8 tables. All IDs are UUIDv7 so they're time-sortable and B-tree friendly.

| Table | PK | Columns worth mentioning |
|---|---|---|
| `users` | uuidv7 | email (unique), first_name, last_name, password_hash |
| `sessions` | uuidv7 | token_hash (unique), expires_at (7 days) |
| `posts` | uuidv7 | content, image_key, is_public, like_count, comment_count |
| `comments` | uuidv7 | post_id FK, content, like_count, reply_count |
| `replies` | uuidv7 | comment_id FK, content, like_count |
| `post_likes` | (post_id, user_id) | |
| `comment_likes` | (comment_id, user_id) | |
| `reply_likes` | (reply_id, user_id) | |

Separate likes tables per content type. No polymorphic junk — each has a real FK to its parent table.

Migrations are in `src/infrastructure/database/migrations/`. Generated by Drizzle.

## Pagination

Cursor-based on every list endpoint. Cursor is `base64url({ createdAt, id })`. Default 20 items, max 100.

There's a composite index on `(created_at, id)` so the query planner seeks straight to the cursor position. Offset would force a full scan of skipped rows.

---

## Denormalized counts

`like_count`, `comment_count`, `reply_count` are plain columns on the parent row, bumped atomically inside the same transaction as the insert or delete. Feed queries never hit `COUNT(*)`.

Top 5 likers uses `ROW_NUMBER() OVER (PARTITION BY ...)` so it's one query, not N+1.

---

## Public / private posts

`posts.is_public` defaults to true.

Feed query: `WHERE is_public = true OR user_id = $currentUserId`.  
Author's own posts endpoint accepts `?includePrivate=true` but only if the viewer owns the profile.  
Fetching a single private post returns 404 — not 403 — so you can't probe whether it exists.  
Likes and likers endpoints also reject hidden posts.

---

## Image uploads

Separate upload module, not crammed into the post/comment endpoints.

1. `POST /api/v1/uploads/images` — multipart, 5MB max, JPEG/PNG/WebP only. Returns `{ key, url }`
2. Pass `imageKey` to the create/update endpoint for posts or comments

Keeps all upload logic in one spot and makes it reusable later (profile pics, covers, whatever).

Keys are `{folder}/{userId}/{uuidv7}.{ext}` — unique, not guessable.

When a post is deleted we fire off an R2 delete in the background. If it fails the user still gets a 200 — not their problem.

---

## Rate limiting

- Register: 2/min
- Login: 10/min
- Everything else: 100/min (env-configurable)

Redis-backed with an in-memory fallback. Uses `@nestjs/throttler`.

---

## Security

- Helmet via `@nestjs/helmet`
- CORS limited to the frontend origin
- Every DTO runs through class-validator — bad input (including SQL injection payloads) gets rejected as invalid format
- Passwords: argon2id, never logged, never serialized to JSON
- Session tokens: SHA-256 hashed before they touch the DB
- httpOnly cookie so XSS can't grab the token
- Private posts: 404, not 403
- `.env` validated on startup with Joi

## Response shape

```json
{
  "success": true,
  "data": { ... },
  "meta": {
    "timestamp": "2024-...",
    "requestId": "uuidv7",
    "pagination": { "nextCursor": "base64url...", "hasMore": true }
  }
}
```

Every request gets an `x-request-id` header (UUIDv7) logged for tracing.

## Setup

```bash
cp .env.example .env
docker compose up -d          # postgres 17 + redis 7
yarn install
yarn db:migrate               # runs drizzle migrations
yarn start:dev                # localhost:4000
```

Swagger at `http://localhost:4000/docs`.

### .env

```
DATABASE_URL=postgres://feedbookuser:feedbook@localhost:5432/feedbookdb
REDIS_HOST=localhost
REDIS_PORT=6379
R2_ACCOUNT_ID=
R2_ACCESS_KEY_ID=
R2_SECRET_ACCESS_KEY=
R2_BUCKET_NAME=
R2_PUBLIC_URL=
PORT=4000
FRONTEND_URL=http://localhost:3000
SESSION_EXPIRY_DAYS=7
```

## Tests

```bash
yarn test
```

4 test suites, all using the NestJS testing harness with mocked repos:

- **auth** — duplicate email gets rejected, wrong password gets rejected, good login returns session + user
- **password-hasher** — same input produces different hashes, verification works both ways
- **posts** — non-owner gets 404 on private post, can't update someone else's post, can't delete someone else's post
- **security** — SQL injection garbage in email field rejected by class-validator before it reaches the controller
